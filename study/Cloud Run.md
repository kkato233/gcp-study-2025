# Cloud Run まとめ

## 概要
- 対象: 完全マネージドのサーバレスコンテナ実行基盤（Knativeベース）
- 主用途: Web/API、イベント駆動、バッチ（ジョブ）、突発トラフィック、運用省力

## 利点
- サーバレス: ゼロスケール、急激なスパイクへの高速オートスケール
- 運用容易: OS/ノード管理不要、リビジョン/ロールバック、最小インスタンスでコールドスタート緩和
- 連携: Cloud Scheduler、Pub/Sub、Eventarc、VPCコネクタ、Cloud SQL/Firestore 等
- セキュリティ: IAM/サービス間認証、リクエスト単位のアイソレーション

## 欠点
- 実行モデル制約: リクエスト駆動（長時間常駐は不向き）、受信上限/同時実行数調整が必要
- 状態管理: ローカルディスク短命、永続は外部ストア依存
- ネットワーク: 高スループット低レイテンシな長時間接続やカスタムネットワーク要件は苦手

## 価格の利点
- 完全従量: vCPU・メモリ・リクエスト課金。アイドル時ほぼゼロ
- フリークエータ: 小規模/開発段階でコストを極小化可能
- 自動スケール: 必要な時だけ動作し費用発生

## 価格の欠点
- 常時稼働: 最小インスタンスを高く設定/常時高負荷だとPaaSより高くなりうる
- egress/周辺: VPCコネクタ/ネットワーク/ログで追加費用が積み上がる

## 選定の目安
- 向く: イベント/HTTP駆動のマイクロサービス、スパイク対応、運用最小化
- 代替: 常時高負荷/長時間プロセス→ GKE/GCE、特殊ドライバ→ GCE

## コスト最適化 Tips
- 最小/最大インスタンスと同時実行数を調整してコールドスタートと費用のバランス
- イメージを軽量化し起動時間短縮、未使用リージョン無効化
- キャッシュ/キューで平準化、ログ/メトリクスの保持期間チューニング
